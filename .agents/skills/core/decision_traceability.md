# 決策追溯與意圖同步技能

## 概述

本技能要求 Agent 只在極少數、使用者明確同意的情況下，才將真正需要保留理由與脈絡的重要決策記錄到 `.agents/decisions/decision_history.md`，並將使用者在任務中確認的目標約束、非目標與整體意圖同步到最適合的既有文件中。

它的目標是讓後續 Agent 能先查過去脈絡，再做當下判斷，同時避免把決策歷史變成流水帳噪音。

## 觸發條件

- 每次開始處理新的使用者任務時。
- 使用者提出新的產品目標、工作限制、流程規範、設計意圖或非目標。
- 任務中確認了會影響未來工作的決策。
- Agent 發現目前做法與過往決策、既有文件中的有效意圖或任務約束可能衝突時。

## 核心規則

- **任務前先查脈絡**：開始規劃前，必須依任務關鍵字搜尋 `.agents/decisions/decision_history.md`，並讀取相關 mission、skill、workflow、architecture、規格或 README。
- **決策歷史極高門檻**：只有當一筆內容能回答「當時至少有兩條合理路徑，為什麼選了其中一條而不選另一條」時，才可能進入決策歷史。
- **使用者明確同意才可新增**：Agent 不得自行新增決策歷史。即使內容看似重要，也必須先詢問使用者；只有使用者明確回答該項目可以被記錄進 `.agents/decisions/decision_history.md`，才可新增。
- **主文件優先**：若新的目標、限制或做事方式能直接在 mission、skill、workflow、architecture 或 README 中約束 Agent，就優先更新該主文件。
- **規則不是決策**：會長期影響產品方向、視覺定位、技術選型、部署、資料邊界、工作流程或文件制度的內容，若只是在建立規則、同步目標或整理已確認方向，應寫入主文件，不應寫入決策歷史。
- **取捨理由才是候選**：只有「主文件能寫清楚規則，但仍不足以解釋未來為何不能改回另一條曾被認真考慮的路徑」時，才是決策歷史候選項。
- **目標約束必須放回適合文件**：若使用者確認新的長期目標、限制、非目標或整體意圖，必須先尋找最貼近語意的既有文件與段落。
- **新檔案需先詢問**：若現有文件都不適合承載新的目標、限制或意圖，必須先詢問使用者是否要建立新文件。
- **決策不是聊天逐字稿**：不要記錄一般討論、臨時想法、修訂過程、單次任務細節、已能由主文件清楚約束的規則，或已撤回方向。
- **不得把主文件摘要搬進決策歷史**：若 mission、architecture、domain skill、workflow 或 AGENTS.md 已能讓後續 Agent 正確行動，禁止再用「方便後續查閱」或「第一版確認」為理由新增決策歷史。
- **保留來源與狀態**：每筆決策應包含日期、狀態、觸發來源、決策內容、理由、影響範圍與後續 Agent 行動。
- **衝突時優先釐清**：若新需求與既有決策或文件中的有效意圖衝突，且使用者沒有明確覆蓋舊決策，Agent 必須指出衝突並詢問。

## 決策歷史新增守衛

Agent 想新增 `.agents/decisions/decision_history.md` 前，必須逐項通過以下守衛。任一項不通過，就不得新增，只能更新主文件或在最終回覆中說明沒有新增決策歷史。

1. **替代路徑守衛**：這筆內容是否明確存在至少兩條曾被認真考慮的合理路徑？若沒有替代路徑，不是決策歷史。
2. **取捨理由守衛**：這筆內容是否能回答「為什麼選 A 而不選 B」？若只能回答「現在規則是什麼」，不是決策歷史。
3. **主文件不足守衛**：mission、skill、workflow、architecture、README 或 AGENTS.md 是否已足以約束後續 Agent？若足夠，不得新增決策歷史。
4. **未來誤判風險守衛**：若沒有這筆決策，未來 Agent 是否很可能因不知道當時取捨而把方案改回錯誤方向？若只是少一段背景，不得新增。
5. **長期有效守衛**：這筆內容是否會長期影響產品、技術、資料、部署、工作流程或風險邊界？若只是本次任務、暫時修正或整理過程，不得新增。
6. **使用者同意守衛**：Agent 是否已經明確詢問使用者，而使用者也明確回答這筆內容可以記錄進決策歷史？若沒有，不得新增。

詢問使用者時，必須提出具體候選內容與理由，而不是泛泛詢問「要不要更新決策歷史」。例如：

> 我認為這可能是決策歷史候選，因為當時在 A 與 B 之間做了長期取捨。是否同意把這筆內容記錄進 `.agents/decisions/decision_history.md`？

只有在使用者明確同意後，Agent 才能新增該筆紀錄。

## 任務前檢查步驟

1. **搜尋決策歷史**：用任務相關關鍵字搜尋 `.agents/decisions/decision_history.md`。
2. **尋找相關文件**：依任務性質讀取 mission、feature domains、product experience、skill、workflow、architecture、規格或 README。
3. **比對目標**：確認當下任務是否與既有決策、相關文件中的有效意圖或專案使命衝突。
4. **建立行動假設**：若無衝突，依相關文件與決策歷史執行；若有不明確處，先詢問使用者。

## 任務中與任務後同步步驟

1. **尋找同步落點**：若任務改變有效中的目標、限制、非目標或整體意圖，先找最適合的既有文件與段落。
2. **主文件先更新**：能由主文件清楚約束後續 Agent 的內容，直接更新主文件。
3. **判斷是否為決策歷史候選**：只有當內容通過「替代路徑、取捨理由、主文件不足、未來誤判風險、長期有效」守衛時，才可視為候選。
4. **新增前必須詢問**：決策歷史候選仍不得直接寫入；必須先詢問使用者，並取得使用者明確同意該項目可以被記錄進 `.agents/decisions/decision_history.md`。
5. **必要時先詢問新文件**：若沒有合適的既有文件，先詢問是否建立新文件與建議位置。
6. **巡查相關文件**：依 `.agents/skills/core/documentation_sync.md` 檢查其他相關文件是否也需要同步。
7. **收尾回報**：最終回覆中簡短說明更新了哪些相關文件；若沒有使用者明確同意新增決策歷史，應說明未新增決策歷史。

## 重要決策判斷範例

- **可能成為候選**：部署方式、核心架構、資料邊界或重大工作流程曾在兩個以上合理方案間取捨，且主文件只寫規則仍不足以避免未來誤改。
- **可能成為候選**：未來若只看到主文件規則，仍可能不知道為何不能改回另一種方案的決策。
- **主文件即可**：新的文案語氣規則、UI 細節偏好、工作步驟或文件同步要求，且寫入對應文件後已足夠約束 Agent。
- **主文件即可**：第一版使命、第一版技術棧、資料邊界、匿名策略、domain skill 正式性或 agent workflow，只要主文件已能清楚規範後續行動。
- **通常不需留檔**：單次錯字修正、一次性排版調整、不影響未來判斷的臨時說明。
- **禁止自行留檔**：看似重要但未取得使用者對「記錄進決策歷史」的明確同意。

- 此技能與 `.agents/skills/core/goal_oriented_design.md` 搭配使用。
- 此技能與 `.agents/skills/core/documentation_sync.md` 搭配使用。
- 若使用者最新明確指示覆蓋過往決策，Agent 可以依新指示更新主文件；若要新增、修訂或保留決策歷史紀錄，仍必須取得使用者對該紀錄的明確同意。
