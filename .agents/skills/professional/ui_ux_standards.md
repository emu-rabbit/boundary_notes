# UI/UX 與視覺規範 (UI/UX Standards)

## 概述

本技能確保 Agent 在建立或修改任何使用者可見介面、文案、互動與視覺資產時，維持清楚、可用、可維護，並對齊本專案已建立的使命與產品方向。

## 觸發條件

- 建立或修改頁面、section、layout、CSS、動畫、字體、色彩、圖片或互動時。
- 撰寫使用者可見文案、導覽、錯誤訊息、說明文字、CTA 或空狀態時。
- 評估介面是否有視覺噪音、資訊層級混亂、可用性問題或 mobile/desktop 品質落差時。

## 核心規則

### 體驗定位

- **使命優先**：本專案使命已建立於 `.agents/mission/project_mission.md`；介面、文案、互動與視覺資產必須對齊該文件。
- **知情同意優先**：秘密檔案、分享連結、測驗流程與結果檢閱都必須提醒使用者這是溝通工具，不是同意書、契約或承諾。
- **教育工具語氣**：BDSM 相關內容應以教育、自我理解與界限核對為導向，避免露骨、挑逗、獵奇或鼓勵使用者嘗試特定項目。
- **兔子陪伴角色**：插畫兔子應呈現可愛、溫柔、陪伴與引導感，可用隱喻表達敏感情境，不得 NSFW 或過度露骨。生成、修改或評估主角兔子圖片時，必須讀取 `.agents/skills/visual/main_rabbit_image_generation.md`，並以 `.agents/assets/characters/main_rabbit_role.png` 作為核心人物設定 reference。
- **非社交產品**：介面不得暗示配對、交友、邀約、排行、競賽或他人評分。
- **使用者目標優先**：畫面應幫助使用者完成任務，不把裝飾或技術展示放在核心流程之前。
- **具體勝過泛泛**：文案應直接、具體、可理解，避免模板化空話與內部術語外露。
- **劇情頁的 RPG 對話體感**：前導劇情播放頁應優先維持單頁 RPG 對話畫面感，以固定場景、角色與對話區切換狀態，不因手機版而退化成傳統長頁閱讀。
- **其他頁面的彈性版面**：主頁、設定、關於與後續功能頁可依內容量選擇單頁式畫面或可滾動畫面；當內容需要閱讀或操作區塊超出一屏時，可以自然出現拉條，但路由切換應從頁面頂端開始，背景與視覺層不得因滾動位置產生不自然放大、拉伸或跳動。

### 視覺設計

- **建立一致系統**：色彩、字體、間距、邊框、陰影、圖案與動效都應有一致規則。
- **暗色調一致性**：本專案視覺應維持 BDSM 的神秘感與教育工具的溫和包容氛圍；UI 背景、前景面板、按鈕、輸入框與裝飾要素應一律往暗色調設計，避免深色背景搭配大面積亮色前景造成氛圍斷裂。
- **自刻 UI**：本專案 UI 應自己刻，不使用現成 Vue UI/UX library 或模板化 component kit；若使用無樣式 helper，仍必須保留本專案自己的視覺語言。
- **可讀性不可犧牲**：文字對比、行高、段落寬度、按鈕大小與 focus 狀態必須可讀可用。
- **裝飾需有目的**：裝飾元素應服務身份、敘事、掃讀或動線；若只增加噪音，應刪除。
- **避免一味套版**：不要無理由複製常見 landing page、dashboard、portfolio 或 SaaS 模板結構。
- **主題不單調**：避免整個介面只依賴單一色相或常見紫藍漸層、米色履歷風、深藍 dashboard 風。

### 版面與響應式

- **手機與桌面同等優先**：不可只完成桌面後讓手機被動縮放，也不可只為手機犧牲桌面資訊密度與視覺節奏。
- **穩定尺寸**：固定格式元素需有明確 `aspect-ratio`、min/max、grid track 或其他約束，避免 hover、載入或文字長度造成 layout shift。
- **文字不可溢出**：任何 viewport 中，標題、按鈕、卡片與標籤文字都不可互相重疊或超出容器。
- **個人化大標題防線**：「{暱稱}的祕密檔案」這類含使用者輸入的大標題，應以共用元件拆成暱稱、語系連接詞與固定產品名片段；名稱長度應以視覺寬度量測並用於決定縮字密度，不使用單純字數一刀切，也不可靜默裁切使用者名稱。畫面呈現優先使用雙行 title，讓暱稱留在大標中、連接詞字級較小、固定產品名不可拆字；在關鍵螢幕寬度可能換行時，依內容長度縮小字體，而不是放任瀏覽器產生不舒適斷句。
- **觸控友善**：行動版互動目標要足夠大，間距要避免誤觸。
- **掃讀效率**：桌面版應保留清楚資訊層級與掃讀路徑，不要把所有資訊壓成手機式單欄。

### 互動與動效

- **微互動要克制**：hover、scroll、cursor、transition 或 canvas 動效應強化理解，不應干擾閱讀。
- **尊重 reduced motion**：動畫需支援 `prefers-reduced-motion` 或提供低動態 fallback。
- **互動可退化**：核心內容與操作在 JavaScript 失效、資源慢載入或 API 失敗時應有合理 fallback。
- **資源預熱服務流程**：圖片、插畫與大型資源的預熱應服務測驗、結果檢閱與分享流程的順暢感，不應為裝飾搶佔初始載入。
- **不要用內部術語當 UI**：使用者可見文案應自然，避免工程狀態名、檔案名或內部分類外露。

### 多語系文案

- **繁體中文原文優先**：繁體中文是使用者手寫原文與產品語氣 source of truth。除標點符號誤用、明確錯字或使用者要求外，不得為了抽 i18n、翻譯、排版或重構改寫既有繁體中文文案。
- **翻譯保留人味**：簡體中文、日文與英文應貼合繁體中文原文的內容、情緒與溫度，但可依語言習慣重組句子；避免機翻感、過度正式或過度露骨。
- **短句優先**：非繁中語系應盡量使用短句與穩定長度，降低按鈕、卡片、標題與 mobile layout 爆版風險；若翻譯長度明顯影響排版，優先微調該語系譯文或 responsive 約束，不改繁中原文。

### 檢查分工

- **Agent 負責機械風險**：Agent 應檢查 build、資產路徑、DOM/CSS 結構、圖片尺寸屬性、lazy/fetch priority、可及性屬性與明顯 overflow/CLS 風險。
- **視覺檢查依任務決定**：若使用者要求視覺驗證，使用可用瀏覽器或截圖工具檢查；若工具受限，清楚回報限制並交付可預覽狀態。
- **依回饋修正**：使用者提供截圖或文字回饋後，Agent 應以該回饋為準修正實際頁面。

## 執行步驟

1. 設計前先讀取 `AGENTS.md`、任務相關 `.agents` 文件與既有 UI/文案；若涉及主角兔子圖片或 prompt，先讀取 `.agents/skills/visual/main_rabbit_image_generation.md`。
2. 先定義使用者目標、資訊層級、主要操作與可能錯誤狀態。
3. 建立或沿用視覺 token：色彩、字體、間距、形狀、動效與資產風格。
4. 同時規劃 mobile、tablet、desktop 的資訊順序與布局。
5. 實作後檢查文字、圖片、互動、focus、hover、reduced motion 與不同 viewport。

## 注意事項

- 若視覺方向需要品牌、圖片、語氣或使用者資料，而 repo 中尚未提供，先用可替換的結構與 placeholder，並清楚標示需要使用者補內容。
- 若本專案未來建立專屬 UI/UX 或品牌文件，本技能應與那些文件同步更新。
