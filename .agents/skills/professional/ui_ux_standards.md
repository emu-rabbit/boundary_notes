# UI/UX 與視覺規範 (UI/UX Standards)

## 概述

本技能確保 Agent 在建立或修改任何使用者可見介面、文案、互動與視覺資產時，維持清楚、可用、可維護，並對齊本專案已建立的使命與產品方向。

## 觸發條件

- 建立或修改頁面、section、layout、CSS、動畫、字體、色彩、圖片或互動時。
- 撰寫使用者可見文案、導覽、錯誤訊息、說明文字、CTA 或空狀態時。
- 評估介面是否有視覺噪音、資訊層級混亂、可用性問題或 mobile/desktop 品質落差時。

## 核心規則

### 體驗定位

- **使命優先**：本專案使命已建立於 `.agents/mission/project_mission.md`；介面、文案、互動與視覺資產必須對齊該文件。
- **知情同意優先**：秘密檔案、分享連結、測驗流程與結果檢閱都必須提醒使用者這是溝通起點，不是同意書、契約或承諾。
- **私密筆記本語氣**：BDSM 相關內容應以自我理解與界限核對為導向，可以具有教育性質，但不得以教育者或教育服務自居；避免露骨、挑逗、獵奇或鼓勵使用者嘗試特定項目。
- **兔子陪伴角色**：插畫兔子應呈現可愛、溫柔、陪伴與引導感，可用隱喻表達敏感情境，不得 NSFW 或過度露骨。生成、修改或評估主角兔子圖片時，必須讀取 `.agents/skills/visual/main_rabbit_image_generation.md`，並以 `.agents/assets/characters/main_rabbit_role.png` 作為核心人物設定 reference。
- **非社交產品**：介面不得暗示配對、交友、邀約、排行、競賽或他人評分。
- **使用者目標優先**：畫面應幫助使用者完成任務，不把裝飾或技術展示放在核心流程之前。
- **具體勝過泛泛**：文案應直接、具體、可理解，避免模板化空話與內部術語外露。
- **劇情頁的 RPG 對話體感**：前導劇情播放頁應優先維持單頁 RPG 對話畫面感，以固定場景、角色與對話區切換狀態，不因手機版而退化成傳統長頁閱讀。
- **建立檔案劇情延續**：建立新檔案的劇情序幕必須沿用前導劇情相同的場景、角色與緊湊對話卡構圖，不得重做成置中資訊頁或拉長對話卡。前導初次相遇的說話者維持「不知哪來的兔子」；只有進入建立測驗的劇情後，才稱呼牠為「守密兔」。劇情分支用的內部標記不得出現在正式文案；非主要操作的小型輔助文字應置中，不干擾主要選項區。
- **時光機比對的減法設計**：前導劇情結束後應直接進入獨立的兩檔差異工作區，不重播結語，也不沿用故事對話構圖。畫面以清楚的較早到較新時間軸交代新檔名稱、必要的舊暱稱、兩份最後編輯時間與 scope，再依「焦點喜好、大分類、絕對禁止底線、其他細項」的固定順序只顯示有變動的區塊；手機首屏不得把返回、重選、檔名、原暱稱、兩個日期、scope 與方向切換拆成逐條散落內容：返回與重選同列，其他身份與時間資訊收進一張無巢狀卡片的檔案摘要，日期維持較早到較新的左右對照，方向切換作為摘要最後一列。焦點喜好、大分類與絕對禁止各先取前 4 筆差異，其他細項因採密集列表可先取前 8 筆，完整長列表再由單一 dialog 承接。四區不可只是同款卡片更換標題或被打平成同一套 2×2：焦點喜好沿用唯讀檔案的暖金語彙成為第一視覺主角，並給第一個變動較大的欄寬與字級；排序依序為成為第 1 位、其他新加入、排行向前、排行向後、被移除，新加入且直接成為第 1 位歸入第一類。暖金只用於區塊標線、一般字重且無陰影的區塊標題與第一筆結果，其餘分類名、順位與邊框回到暖白／玫瑰色，避免整區同時發亮。大分類維持完整且平衡的標準分類卡；同列卡片因沒有備註或只改變較少欄位而內容較短時，內部比較內容仍一律從上緣開始，不用垂直置中破壞共同掃讀基準線。大分類卡左側的分類名稱是唯一標題，右側不得再重複「分類整體」，桌機、窄版與完整 dialog 均一致；只有無法取得歷史分類名稱時才保留題名 fallback。完整 dialog 的大分類卡改以上方小圖加分類名的橫列先建立辨識，再讓下一列狀態面使用完整寬度；背景只在分類圖附近保留局部襯托，不使用指向空白中央的聚光。單張或奇數末張的滿寬大分類卡不可因欄位數量退回多條左右 diff；應沿用單一中央方向與較早／較新整組狀態面，未變欄位直接省略，只有一個主要欄位時由該欄位撐滿狀態面。備註在狀態面底部以分隔線後的低視覺重量文字呈現，不與經驗、喜好爭奪焦點；狀態面僅有備註時，必須移除失去分隔對象的線條與間距。絕對禁止以克制的莓紅底與實體邊界強調界線，內容固定保留「新列為絕對禁止」與「已移出絕對禁止」兩欄，每筆只保留分類圖、項目名與偏好值到 `hardNo` 或自 `hardNo` 移出的方向；其中一欄沒有內容時以低視覺重量的「無」保留閱讀座標，不讓欄位消失，即使經驗或備註同時改變也不在此區顯示。其他細項使用滿寬、較小圖面、低陰影與緊密分隔線的一列列核對節奏，每題只保留一個中央箭頭，把經驗、喜好與備註分別收進較早與較新的整組狀態面，避免同一橫列出現多條競爭的閱讀方向。分類圖是一個分類群組的主要辨識面：一張足以辨識的 1:1 圖搭配一次分類名，統領該分類的所有變動；不得退化成區塊標題旁的小縮圖，也不得在逐筆內容重複圖片與分類名。完整 dialog 沿用各區自身的構圖與分類分組。題目只有在兩檔皆已作答時才可列入比較；scope 新增或移除的整側只顯示一次先前／之後未有作答紀錄，不得展開成逐題差異。不得用總變動數、檔案 ID、版本標籤、重複方向文字、逐區無變動狀態或框中再包框的報表層次增加掃讀負擔；四區皆空且沒有整側提示時才顯示一次整體空狀態。選檔清單每列只保留名稱、必要來源、scope 與日期的清楚層級；同來源不得逐列重複，重新開啟 dialog 時需回到目前選項或清單頂端，不得露出被裁切的半張選項。
- **時光機絕對禁止預覽計數**：「新列為絕對禁止」與「已移出絕對禁止」兩欄各自獨立套用前 4 筆預覽上限，不得先合併兩欄再計數；完整變動入口也只在任一欄各自超過上限時顯示。
- **高重複作答流程**：分類與細項作答屬於數十至數百次的高重複流程，版面應以單頁可見為優先，並保有明確的目前位置與返回上一題路徑；不必為了顯示百分比而額外增加進度條。每題應保留分類視覺錨點、避免重複呈現同一方向語意，且把必填選項做成可快速辨識的圖示加短標籤；兩項必填答案完成後應立即保存，保留清楚的「下一題」按鈕供使用者立即繼續，也可在短暫確認後自動前往下一題。
- **其他頁面的彈性版面**：主頁、設定、關於與後續功能頁可依內容量選擇單頁式畫面或可滾動畫面；當內容需要閱讀或操作區塊超出一屏時，可以自然出現拉條，但路由切換應從頁面頂端開始，背景與視覺層不得因滾動位置產生不自然放大、拉伸或跳動。
- **關於頁的收斂原則**：關於頁不是長型 landing page，桌面版應優先在接近一屏內完成主要內容與必要入口。專案使命、角色定位與產品界線應合併成一次清楚說明，不得再以多組卡片、不同標題或正反清單重複轉述；重播前導可疊放於角色插畫，返回入口使用頁面上方角落的低視覺重量慣例位置。
- **版本歷史的資訊層級**：版本歷史頁由頁面頂端開始排列，標題區只保留 Boundary Notes 品牌小標與頁面標題，不重複顯示動態產品名稱或其他小標。目前版本及其內容使用高對比醒目色；只有較舊版本才使用淡化色彩，不能讓背景遮罩壓低主要內容對比。
- **法律文件閱讀邊界**：使用條款與隱私權政策從產品頁以新分頁開啟，使用單頁靜態閱讀版面且不放返回鍵；法律頁仍要能切換完整四語內容，並以繁體中文版本為準。法律頁不得顯示分析同意 banner，以免遮擋正式內容或讓使用者誤解為閱讀條件。

### 視覺設計

- **建立一致系統**：色彩、字體、間距、邊框、陰影、圖案與動效都應有一致規則。
- **語系字型分流**：預設字型以 Google Fonts 的 `Huninn / jf open 粉圓` WOFF2 unicode-range 子集服務繁體中文、英文與使用者自行輸入的稱呼或文字；不得只依目前固定文案自製 glyph subset，以免未收錄的使用者輸入改用不一致字型。簡體中文與日文不得假設粉圓體完整覆蓋，應透過 `html:lang(...)` 切換到風格接近、開源且 CJK 覆蓋較完整的字型 stack。目前簡體中文使用 Noto Sans SC / Source Han Sans SC 系列優先，日文使用 Noto Sans JP / Source Han Sans JP 系列優先，並保留各平台系統字型 fallback。外部字型必須使用 `display=swap`，讓核心內容不因等待字型而不可讀。
- **CSS 分層一致性**：自訂 CSS 以 `src/styles/` 為維護入口；新增或調整樣式時，依責任放入 foundation、route-shell、story-stage、story-dialogue、home-page、secondary-pages、questionnaire、secret-file-preview 或 responsive，而不是回到單一巨型 stylesheet。建立檔案、分類作答與結果編輯器由 `questionnaire.css` 管理；唯讀結果總覽、焦點排名、分類詳情與閱覽 dialog 由 `secret-file-preview.css` 管理；`responsive.css` 是最後 cascade 層，跨頁 mobile/desktop override 應集中在此檔或與此檔同等順序的位置。
- **暗色調一致性**：本專案視覺應維持 BDSM 的神秘感與私密筆記本的溫和包容氛圍；UI 背景、前景面板、按鈕、輸入框與裝飾要素應一律往暗色調設計，避免深色背景搭配大面積亮色前景造成氛圍斷裂。
- **自刻 UI**：本專案 UI 應自己刻，不使用現成 Vue UI/UX library 或模板化 component kit；若使用無樣式 helper，仍必須保留本專案自己的視覺語言。
- **可讀性不可犧牲**：文字對比、行高、段落寬度、按鈕大小與 focus 狀態必須可讀可用。
- **裝飾需有目的**：裝飾元素應服務身份、敘事、掃讀或動線；若只增加噪音，應刪除。
- **避免一味套版**：不要無理由複製常見 landing page、dashboard、portfolio 或 SaaS 模板結構。
- **主題不單調**：避免整個介面只依賴單一色相或常見紫藍漸層、米色履歷風、深藍 dashboard 風。

### 版面與響應式

- **手機與桌面同等優先**：不可只完成桌面後讓手機被動縮放，也不可只為手機犧牲桌面資訊密度與視覺節奏。
- **單一頁面捲動責任**：一般 route 不應各自建立第二層垂直捲動容器。預設由 `body` 承擔頁面級捲動；route section 與 app shell 以 `min-height` 填滿 viewport，內容不足時不出現拉條，內容在極端短高 viewport 超出時自然撐開文件並可捲到底。
- **避免高度裁切捷徑**：不要用外層 `overflow: hidden`、固定 `height: 100vh/100dvh`、或只針對單一頁面的 `max-height` workaround 來掩蓋 overflow。若某頁需要內部捲動，必須有明確的互動理由，例如面板內長文區，且不能讓主要操作被裁切或形成互相競爭的雙重拉條。
- **背景與黑幕不參與內容高度**：背景圖、黑幕、ambient overlay 與裝飾性偽元素必須與內容層分離，避免因 absolute layer、負 inset、blur、scale 或 mask 造成假高度、底部露白、黑幕斷層或高度足夠卻出現拉條。這類層通常應固定於 viewport、不可接收 pointer event，並由內容層自己的 `min-height` 決定頁面高度。
- **避免 cascade 補丁化**：若一個視覺變更需要同時改多處 selector、提高 specificity、或靠動畫結束狀態覆蓋 base style，先停下來檢查 owner 檔案與 import order。應把 base style、狀態 style、responsive override 分別放在可預期的位置，讓單一元素的維護面積保持小而可讀。
- **穩定尺寸**：固定格式元素需有明確 `aspect-ratio`、min/max、grid track 或其他約束，避免 hover、載入或文字長度造成 layout shift。
- **文字不可溢出**：任何 viewport 中，標題、按鈕、卡片與標籤文字都不可互相重疊或超出容器。
- **個人化大標題防線**：「{暱稱}的祕密檔案」這類含使用者輸入的大標題，應以共用元件拆成暱稱、語系連接詞與固定產品名片段；名稱長度應以視覺寬度量測並用於決定縮字密度，不使用單純字數一刀切，也不可靜默裁切使用者名稱。畫面呈現優先使用雙行 title，讓暱稱留在大標中、連接詞字級較小、固定產品名不可拆字；在關鍵螢幕寬度可能換行時，依內容長度縮小字體，而不是放任瀏覽器產生不舒適斷句。
- **觸控友善**：行動版互動目標要足夠大，間距要避免誤觸。
- **掃讀效率**：桌面版應保留清楚資訊層級與掃讀路徑，不要把所有資訊壓成手機式單欄。
- **方形分類圖完整性**：結果頁使用的 1:1 分類圖必須保留主要物件完整可辨識；總覽、焦點喜好與分類詳情在桌機和手機都應以 `contain` 或等價構圖呈現，不得因卡片被備註撐高就把圖片同步拉高後大幅裁切。
- **唯讀詳情專注模式**：進入唯讀分類詳情後，隱藏總覽用的檔案身份、警語、語系與操作側欄，只保留返回、分類摘要與細項答案。桌機用左摘要、右返回與列表維持核對效率；手機依返回、摘要、列表的自然順序排列。不要用「細項結果／分類細項」等重複標題製造雜音。
- **唯讀語系工具層級**：語系切換只存在於唯讀總覽內容欄上方的低視覺重量工具列，不放進已承載身份、警語、方向與檔案操作的摘要側欄，也不與焦點喜好或分類標題競爭。控制項平時使用弱邊界與次要文字色，hover／focus 時才提高辨識度，但必須保留明確文字標籤讓有需要的閱讀者找得到。
- **唯讀來源對應 CTA**：唯讀總覽的「建立我的檔案」是雲端檔案觀看者的主要導流 CTA，只在 `source=cloud` 時顯示，且必須是該頁最醒目的按鈕；手機版比照編輯結果頁固定於 viewport 底部並預留不遮擋內容的安全空間。本地唯讀預覽不顯示這顆 CTA。分享操作不放在檔案摘要區，應精簡為閱覽語言工具附近、具可及名稱的小型 icon；贊助入口維持摘要區既有的低視覺重量。

### 互動與動效

- **微互動要克制**：hover、scroll、cursor、transition 或 canvas 動效應強化理解，不應干擾閱讀。
- **尊重 reduced motion**：動畫需支援 `prefers-reduced-motion` 或提供低動態 fallback。
- **互動可退化**：核心內容與操作在 JavaScript 失效、資源慢載入或 API 失敗時應有合理 fallback。
- **資源預熱服務流程**：圖片、插畫與大型資源的預熱應服務測驗、結果檢閱與分享流程的順暢感，不應為裝飾搶佔初始載入。
- **不要用內部術語當 UI**：使用者可見文案應自然，避免工程狀態名、檔案名或內部分類外露。

### 多語系文案

- **繁體中文原文優先**：繁體中文是使用者手寫原文與產品語氣 source of truth。除標點符號誤用、明確錯字或使用者要求外，不得為了抽 i18n、翻譯、排版或重構改寫既有繁體中文文案。
- **守密兔稱呼唯一**：正式角色名在繁體中文與簡體中文皆使用「守密兔」、日文使用「守秘うさぎ」、英文使用「Secret-Keeping Bunny」，並以 `src/app/i18n/terminology.ts` 為 runtime source of truth。前導初遇的「不知哪來的兔子」或語境中刻意使用的一般兔子稱呼可以保留，但不得把正式角色名另譯為 `confidentiality rabbit`、`keeper bunny`、`Secretkeeper Bunny` 或其他變體。
- **翻譯保留人味**：簡體中文、日文與英文應貼合繁體中文原文的內容、情緒與溫度，但可依語言習慣重組句子；避免機翻感、過度正式或過度露骨。
- **短句優先**：非繁中語系應盡量使用短句與穩定長度，降低按鈕、卡片、標題與 mobile layout 爆版風險；若翻譯長度明顯影響排版，優先微調該語系譯文或 responsive 約束，不改繁中原文。

### 檢查分工

- **Agent 負責機械風險**：Agent 應檢查 build、資產路徑、DOM/CSS 結構、圖片尺寸屬性、lazy/fetch priority、可及性屬性與明顯 overflow/CLS 風險。
- **CSS 重構驗證**：若目標是不改變畫面或行為的 CSS 重構，應優先比對 production CSS bundle、資產 URL、UTF-8 without BOM、主要 viewport computed metrics 與 browser surface。若 bundle hash 或 computed metrics 變化，需能說明是刻意變更，否則應修正到一致。
- **overflow 實測責任**：修改 route shell、背景、主要 layout 或 responsive CSS 後，應用瀏覽器或等價工具實測一般桌面、極端短高桌面與手機 viewport。檢查重點包含：高度足夠時沒有頁面拉條；高度不足時可以捲到下方內容；背景、黑幕與 ambient overlay 在頂部與底部都沒有高度落差；不存在 route 內外雙重垂直拉條。
- **視覺檢查依任務決定**：若使用者要求視覺驗證，使用可用瀏覽器或截圖工具檢查；若工具受限，清楚回報限制並交付可預覽狀態。
- **依回饋修正**：使用者提供截圖或文字回饋後，Agent 應以該回饋為準修正實際頁面。

## 執行步驟

1. 設計前先讀取 `AGENTS.md`、任務相關 `.agents` 文件與既有 UI/文案；若涉及主角兔子圖片或 prompt，先讀取 `.agents/skills/visual/main_rabbit_image_generation.md`。
2. 先定義使用者目標、資訊層級、主要操作與可能錯誤狀態。
3. 建立或沿用視覺 token：色彩、字體、間距、形狀、動效與資產風格。
4. 同時規劃 mobile、tablet、desktop 的資訊順序與布局。
5. 實作後檢查文字、圖片、互動、focus、hover、reduced motion 與不同 viewport。
