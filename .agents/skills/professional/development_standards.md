# 開發實作規範 (Development Standards)

## 概述

本技能定義本專案的開發基準：維持可讀、可驗證、可維護，並避免在使命與架構尚未明確前過度設計。

## 觸發條件

- 撰寫、修改或重構程式碼、設定、建置流程、測試或資料結構時。
- 選擇是否導入 framework、bundler、套件、外部服務或新工具時。
- 進行 code review、效能檢查或交付前驗證時。

## 核心規則

### 技術選型

- **既有優先**：優先沿用 repo 既有語言、框架、檔案結構、命名與工具鏈。
- **輕量優先**：不得為了簡單需求導入沉重 framework、runtime、大型 UI kit 或外部服務。
- **依賴需有理由**：新增套件前，先確認它解決的問題是否值得增加維護、效能與供應鏈成本。
- **可回退設計**：在專案使命尚未完整定義時，優先選擇局部、低風險、可替換的實作。
- **不移植錯誤脈絡**：不得把 sibling repo 的技術棧、部署方式或產品假設當成本專案預設。

### 程式碼撰寫

- **清楚邊界**：把資料、邏輯、UI、IO、設定與測試維持在清楚的責任邊界內。
- **命名一致性**：遵循既有命名慣例；若尚無慣例，選擇直覺、可搜尋、能表達用途的名稱。
- **錯誤處理**：對外部資源、檔案 IO、使用者輸入、瀏覽器 API、網路或平台差異要有合理 fallback。
- **避免過度抽象**：只有當重複或複雜度真實存在時才新增抽象。
- **註解克制**：只在程式意圖、取捨或非顯而易見的細節需要保留時寫註解。

### 測試與驗證

- **先找現有指令**：執行驗證前，先檢查 `package.json`、README、CI 設定、腳本或既有文件。
- **風險對應驗證**：小型文件變更可用閱讀與 diff 檢查；程式邏輯、建置、資料或使用者可見行為變更應執行對應 test、lint、typecheck 或 build。
- **機械檢查責任**：UI 或資產變更時，Agent 應檢查 build、typecheck、資產路徑、DOM/CSS 結構、可及性屬性與明顯 overflow/CLS 風險。
- **工具失敗要分類**：區分 sandbox、權限、環境、網路與真正程式錯誤；不要把工具權限問題誤判成程式失敗。
- **未能驗證需回報**：若驗證指令不存在或因權限/環境失敗，最終回覆需清楚說明。

## 執行步驟

1. 實作前先讀取 `AGENTS.md`、相關 `.agents` 文件與任務相關程式碼。
2. 判斷是否真的需要新工具、套件、框架或架構；沒有明確理由時維持簡單。
3. 建立或修改程式碼時，同時考慮可讀性、效能、可維護性、錯誤處理與驗證方式。
4. 完成後執行可用且符合風險的 build、lint、test、typecheck 或其他機械檢查。
5. 任務結束前依 `documentation_sync.md` 檢查文件是否需要同步。

## 注意事項

- 如果使用者明確要求特定框架、架構或工具，可採用，但仍需保持範圍精準並同步文件。
- 既有程式碼若不符合本規範，先完成當前工作；非必要重構需另外確認。
